# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) - do not edit manually.

- **[gotcha:project]** `border-border` is NOT the correct design token for this project. Always use `border-edge/<opacity>` (e.g. `border-edge/40`, `border-edge/50`, `border-edge/60`) for structural dividers. `border-border` exists but produces visually inconsistent results in the design system. Check `tailwind.config.ts` - `edge` is the semantic separator token.
- **[pattern:project]** When doing a codebase-wide Tailwind token migration with `sed -i ''` on macOS, sweep from most-specific patterns first (e.g. `border-border/70`, `border-border/60`) down to bare `\bborder-border\b` to avoid double-replacement. Use `grep -rn --include="*.tsx"` for verification passes between sweeps.
- **[fix:project]** For `sed` replacements across multiple known files on macOS, pass all file paths as separate arguments to a single `sed` call: `sed -i '' 's/.../.../' file1 file2 file3`. A `for` loop with `$f` inside a here-string or tool description field silently fails because the variable isn't expanded - the literal `$f` becomes the filename.
- **[gotcha:project]** Paths containing Next.js route groups (parentheses like `(protected)`) need quoting in shell commands: wrap the full path in double quotes, or escape the parens as `\(protected\)` when using `find`/`grep` without `-E`.
- **[gotcha:project]** The `DataTableToolbar` component (`src/components/dashboard/data-table-toolbar.tsx`) renders inside a `Card` immediately after `CardHeader` in `data-table.tsx`. If both carry a `border-b`, two horizontal lines appear in the table header area. The toolbar div should NOT have its own `border-b` - only the `CardHeader` (in `data-table.tsx`) owns that separator.
- **[pattern:project]** Design-system shadow tokens: use `shadow-card` for cards/panels, `shadow-field` for selected-state interactive tiles, and explicit `shadow-[0_3px_0_rgba(0,0,0,0.12),0_6px_16px_rgba(0,0,0,0.08)]` for floating/dropdown surfaces. Avoid generic `shadow-sm`, `shadow-md`, `shadow-lg` - they don't align with the project's visual language.
- **[gotcha:project]** `tests/place-report-merge.test.mts` has 2 pre-existing tsc errors (missing `final_comments` property in test fixtures). Filter them out with `| grep -v "place-report-merge"` when checking tsc output to avoid false alarms - these are not regressions from your changes.
- **[gotcha:project]** Running `tsc --noEmit` without a prior `pnpm build` can produce phantom "file not found" errors for `.next/types/**/*.ts` route stubs if the `.next/` cache is stale or missing. These are benign - focus on non-`.next/types` errors, or run `pnpm build` first to regenerate the stubs.
- **[fix:project]** Uncommenting multi-line JSX blocks wrapped in `{/* ... */}` is best done with two Edit tool calls (remove the opening `{/* <Component` wrapper, then fix the closing `/> */}` separately) rather than perl/sed regex. Perl/sed applied to JSX with special characters (braces, parens, slashes) silently fails or corrupts the output.
- **[gotcha:project]** On macOS, `cat -A` is not a valid flag (it's Linux-only). Use `cat -t -e` instead to show whitespace markers and end-of-line characters.
- **[env:project]** `node ./scripts/verify_audit_contract.mjs` requires the backend repo to be present at `../audit-tools-backend/` with specific instrument JSON files. It fails with ENOENT if run in isolation or if the backend repo is not co-located at the expected relative path - skip this check in frontend-only sessions.
- **[gotcha:project]** Glob patterns that start with `src/` (e.g. `src/components/**/*.tsx`, `src/app/**`) consistently return no results even when the files exist. Always use `**/<filename>` or `**/<pattern>` anchored from `**` instead - e.g. `**/page.tsx` instead of `src/app/**/page.tsx`.
- **[gotcha:project]** The `librarian` subagent type fails immediately with `ProviderModelNotFoundError` in this environment. Only the `explore` subagent type works reliably for background codebase tasks. Do not dispatch `librarian` agents.
- **[pattern:project]** Public/marketing pages (non-authenticated) live outside the `(protected)` route group. The `resources/` route group (`src/app/resources/`) is the existing template: it has its own `layout.tsx` wrapping a shell component and plain `page.tsx` files with explicit `export const metadata` exports. New public/marketing pages should follow this same pattern.
- **[gotcha:project]** Glob with broad wildcard patterns like `**/*.tsx` or `**/*.{ts,tsx}` WITHOUT a `path` parameter returns thousands of node_modules results and is unusable. Always pair broad wildcards with a `path` set to the project root, or use specific filename-anchored patterns (`**/page.tsx`, `**/layout.tsx`) that won't match dependency files at all.
- **[pattern:project]** When Glob patterns are unreliable for project-wide file discovery, use `find . -path './node_modules' -prune -o -path './.next' -prune -o -type f | sed 's#^./##' | sort | head -n 400` via Bash from the project root. This is the most reliable fallback for listing all source files without noise.
- **[gotcha:project]** The next-intl setup does NOT use URL-based locale routing. There is no `defineRouting`, no `localePrefix`, and no `[locale]` URL segments. Locale is resolved from the `playspace_locale_preference` cookie + `Accept-Language` header in `src/i18n/server-locale.ts`. Do not add URL-based locale routing or `[locale]` route group wrappers.
- **[gotcha:project]** Public pages (landing page at `src/app/page.tsx`, resources at `src/app/resources/`) are intentionally NOT internationalized - they use hardcoded English strings and do NOT import `useTranslations` or `getTranslations`. Only `(protected)` pages use next-intl. Do not add `useTranslations` to public marketing or legal pages.
- **[gotcha:project]** `messages/` only contains `en.json` and `de.json` even though `src/i18n/config.ts` declares 5 supported locales (en, de, fr, ja, hi). `loadLocaleMessages` silently falls back to English for any missing locale file. Do not assume all 5 locale files exist.
- **[pattern:project]** Marketing/product images for the landing page live in `public/marketing/`. Screenshot infrastructure for docs/visual testing writes to `public/screenshots/` (run via `pnpm screenshots:web`). Keep these two directories separate.
- **[gotcha:project]** The asset index (`asset-index.json`) lives in the sibling `../assets/` repo at `../assets/asset-index.json`, NOT in `public/`. `scripts/build-asset-index.mjs` reads from `../assets/` and writes back there. Do not look for or create `public/asset-index.json` - that path is no longer used.
- **[gotcha:project]** When using next-intl's `t()` with a computed/dynamic key (e.g., template literal `` `filters.${value}` ``), TypeScript rejects it because the typed signature only accepts literal keys. Workaround: `const tLabel = t as unknown as (key: string) => string;` then call `tLabel(dynamicKey)`.
- **[pattern:project]** Cloudinary signed uploads use `CldUploadWidget` from `next-cloudinary` with `signatureEndpoint="/api/sign-cloudinary-params"`. The signing route imports `import { v2 as cloudinary } from "cloudinary"` (not `next-cloudinary`) and calls `cloudinary.utils.api_sign_request(paramsToSign, apiSecret)` - this keeps `CLOUDINARY_API_SECRET` server-side only and never reaches the browser.
- **[pattern:project]** For Cloudinary image variants delivering UI screenshots, use `q_auto:good` instead of plain `q_auto` for the `full` variant. Plain `q_auto` over-compresses sharp edges and text in screenshots. Size `full` at `w_2880` (not `w_1440`) to stay crisp on 3× retina displays rendering in up to ~960px CSS slots.
- **[gotcha:project]** `playspace.ts` uses a namespaced singleton object (`playspaceApi.admin.*`, `playspaceApi.manager.*`, etc.), NOT top-level `export async function` declarations. Searching for `export async function` returns almost nothing useful. To explore API method patterns, grep for namespace keys like `admin:`, `overview:`, `list:` or read the object structure directly starting around line 700.
- **[convention:project]** All Zod schemas and inferred TypeScript types for new API domains go in `src/lib/api/playspace-types.ts`. The actual API call implementations (using `fetchValidatedJson`/`fetchNoContent`) go in the namespaced object in `src/lib/api/playspace.ts`, which imports from `playspace-types.ts`. Both files must be updated together when adding a new API domain.
- **[gotcha:project]** The Cloudinary `CldUploadWidget` portals to `document.body`, outside any Radix `Dialog`. When used inside a modal dialog, the dialog's focus trap and `pointer-events` lock on `body` makes the widget unclickable (it appears but freezes). Fix: track widget open state with `onOpen`/`onClose` callbacks, then pass `modal={!widgetOpen}` to the wrapping `<Dialog>` and guard `onInteractOutside` to prevent the form closing when the user clicks inside the widget overlay.
- **[pattern:project]** To verify i18n key parity between `en.json` and `de.json` for a specific namespace after adding or removing keys, use: `jq -r '.<namespace>|paths|join(".")' messages/en.json | sort > /tmp/en.txt; jq -r '.<namespace>|paths|join(".")' messages/de.json | sort > /tmp/de.txt; diff /tmp/en.txt /tmp/de.txt && echo "PARITY OK"`. Run this after any i18n changes to catch mismatches before committing.
- **[pattern:project]** Use `FilterPopover` (`src/components/dashboard/filter-popover.tsx`) for multi-select filter UX in dashboard tables - it accepts `title`, `options`, `selectedValues`, and `onChange: (values: string[]) => void`. Prefer it over bespoke single-select `<Select>` dropdowns when building new filterable tables, for consistent UX with the rest of the admin surfaces.
- **[pattern:project]** The admin sidebar nav is defined in `getNavItems()` in `src/components/app/app-shell.tsx` (around line 62). Feature-flagged nav items use the spread pattern: `...(isBugReportingEnabled() ? [{ label: t("bugReports"), href: "/admin/bug-reports", icon: Bug }] : [])`. Follow this pattern for any new conditionally-visible nav entry.
